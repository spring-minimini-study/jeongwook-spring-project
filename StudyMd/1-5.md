## 1. 프레임워크 기초 및 아키텍처

### 🛠 Spring vs Spring Boot
- **Spring**: 자바 엔터프라이즈 개발을 위한 거대 프레임워크. 설정이 세밀하지만 복잡함.
- **Spring Boot**: 스프링을 기반으로 '자동 설정(Auto Config)' 기능을 추가하여 생산성을 극대화한 도구.

### 🌐 웹 애플리케이션 방식
- **MPA (Multi Page Application)**: 서버에서 HTML을 만들어 응답 (전통적인 방식).
- **SPA (Single Page Application)**: 서버에서 데이터(JSON)만 받고, 브라우저의 JS가 화면을 동적으로 그림.

### 🏗 계층형 아키텍처 (Layered Architecture)
사용자의 요청부터 데이터 저장까지 역할을 나누어 관리합니다.
1. **Controller**: 요청 수신 및 전체 흐름 제어.
2. **Service**: 비즈니스 로직(업무 규칙) 처리.
3. **Repository**: DB 접속 및 데이터 조회/저장.
4. **Entity**: DB 테이블 규격과 매핑되는 데이터 객체.

**🔄 데이터 흐름**: `브라우저` ➡️ `Controller` ➡️ `Service` ➡️ `Repository` ➡️ `DB (Entity 활용)`

## 2. DI (Dependency Injection)와 빈(Bean)

### 💡 DI (의존성 주입)
- **정의**: 사용할 객체를 직접 `new`로 만들지 않고, **외부(Spring)에서 생성하여 전달**받는 방식.
- **장점**: 부품(구상 클래스) 교체가 쉽고 테스트가 용이함.

### 📦 Bean (빈) & DI 컨테이너
- **Bean**: 스프링이 직접 생성하고 관리하는 자바 객체. (기능을 담당하는 '도구' 위주로 등록)
- **DI 컨테이너 (ApplicationContext)**: Bean들을 담아두고 조립해 주는 상자.
- **장점**:
    - **싱글톤(Singleton)** 보장 (메모리 절약)
    - **트랜잭션/보안** 등 스프링의 부가 서비스 지원

## 3. 컴포넌트 스캔과 빈 등록

### 🔍 컴포넌트 스캔 (@ComponentScan)
설정 클래스(`JavaConfig`)에 붙이면 하위 패키지를 훑어 **스테레오타입 애너테이션**이 붙은 클래스를 찾아 자동으로 Bean으로 등록합니다.

### 🏷 스테레오타입 애너테이션
| 애너테이션 | 적용 대상 | 특징 |
| :--- | :--- | :--- |
| **@Controller** | 컨트롤러 클래스 | 웹 요청 처리 기능 제공 |
| **@Service** | 서비스 클래스 | 비즈니스 로직 담당 (특별한 부가기능 없음) |
| **@Repository** | 리포지토리 클래스 | DB 예외를 스프링 예외로 변환하는 기능 포함 |
| **@Component** | 일반 공통 부품 | 위 항목에 해당하지 않는 범용 객체 |

```java
// 예시: 구상 클래스에 스티커 붙이기
@Service
public class TrainingServiceImpl implements TrainingService {
    // ... 로직
}
```
## 4. @Autowired를 이용한 의존성 주입

### 🧲 주입 방식 3가지
스프링은 빈을 조립하기 위해 세 가지 방식을 지원합니다.

1. **생성자 주입 (가장 권장)**: 생성자를 통해 의존 객체를 전달받습니다. `final` 키워드를 사용하여 객체의 불변성을 보장할 수 있으며, 테스트 코드 작성 시 유리합니다.
2. **Setter 주입**: `set` 메서드를 생성하여 주입합니다. 주입받는 객체가 변경될 가능성이 있는 경우 사용합니다.
3. **필드 주입**: 변수에 직접 `@Autowired`를 붙입니다. 코드가 간결하지만 외부에서 변경이 어렵고 스프링 컨테이너 없이는 테스트가 힘들어 실무에서는 지양합니다.

```java
// ✅ 실무 표준: 생성자 인젝션 예시
@Service
public class TrainingServiceImpl implements TrainingService {
    // 1. final을 붙여 변경 불가능하게 설정
    private final TrainingRepository trainingRepository;

    // 2. 생성자에 @Autowired 부착 (생성자가 1개면 생략 가능)
    @Autowired 
    public TrainingServiceImpl(TrainingRepository trainingRepository) {
        this.trainingRepository = trainingRepository;
    }
}
```

## 5. 빈(Bean) 충돌 해결 (우선순위)

인터페이스(`TrainingRepo`)를 구현한 구상 클래스가 여러 개(예: `JdbcRepo`, `MockRepo`)일 때, 스프링은 아래의 3단계 우선순위에 따라 주입할 빈을 결정합니다.

1. **@Primary**: 여러 후보 중 "기본 대장"을 지정합니다. 가장 먼저 선택되는 1순위입니다.
2. **@Qualifier("이름")**: 명찰을 붙이는 것과 같습니다. 특정 이름을 명시하여 어떤 부품을 가져올지 정확히 지시합니다.
3. **변수명 매칭**: 파라미터나 필드의 변수 이름이 빈의 이름(클래스명의 첫 글자를 소문자로 바꾼 것)과 같으면 해당 빈을 주입합니다.

## 6. DI 컨테이너 구동 및 인젝션 방법

### 🏗 DI 컨테이너 직접 구동 (ApplicationContext)
코드에서 직접 DI 컨테이너를 생성하고 관리되는 빈을 꺼내 쓸 수 있습니다. 주로 `JavaConfig` 설정 정보를 사용합니다.

```java
// 1. DI 컨테이너 생성 (ShoppingApplication.class는 JavaConfig 클래스)
ApplicationContext context = new AnnotationConfigApplicationContext(ShoppingApplication.class);

// 2. getBean()을 통해 관리되는 객체(Bean)를 반환받음
TrainingService trainingService = context.getBean(TrainingService.class);

// 3. 기능 실행
trainingService.findAll();
