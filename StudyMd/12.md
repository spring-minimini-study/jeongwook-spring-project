## 📚 Spring Framework 학습 기록 (1부: 12장 - RESTful 웹 서비스)

### 핵심 주제: 화면이 아닌 데이터를 주고받는 방식과 REST 가이드라인의 이해

---

### 1. 현대 웹의 흐름: SPA와 마이크로서비스

과거에는 서버가 화면을 다 만들어서 보내줬다면, 이제는 역할이 분담되었습니다.

**SPA (Single Page Application)**  
- **비유**: 한 번 메뉴판(HTML)을 받으면, 그 후로는 음식(데이터)만 계속 주문해서 내 자리에서 그릇을 채우는 방식입니다.
- **장점**: 화면 전체를 새로고침하지 않아도 되니 사용자 경험이 아주 부드럽습니다.  
  예시: 인스타그램, 지메일

**마이크로서비스 (Microservices)**  
- **비유**: 거대한 뷔페 하나를 운영하는 게 아니라, 떡볶이집, 피자집, 일식집을 각각 따로 운영하는 것입니다.  
- **장점**: 한 곳이 망가져도 다른 곳은 멀쩡하고, 각자 제일 잘하는 기술을 쓸 수 있습니다.

### 2. REST란 무엇인가? (Representational State Transfer)

REST는 웹 서비스를 만들 때 지켜야 할 **'매너'** 혹은 **'가이드라인'**입니다. 이 매너를 잘 지킨 서비스를 **RESTful 웹 서비스**라고 부릅니다.

#### 📜 REST의 7가지 황금 규칙
- **URL로 리소스 식별**: 모든 자원(상품, 사용자 등)은 고유한 주소(URL)를 가져야 합니다.  
  예시: `/product/p1`
  
- **HTTP 메서드로 조작**: 행위(조회, 등록, 수정, 삭제)는 주소에 적지 말고 HTTP 메서드(GET, POST 등)로 표현합니다.
  
- **응답 바디 형식**: 데이터는 주로 JSON이나 XML 형식을 사용합니다.
  
- **요청 바디 형식**: 데이터를 보낼 때도 형식을 지정하고 헤더에 명시합니다.
  
- **상태 코드 활용**: 성공했는지, 실패했는지 숫자로 명확히 알려줍니다.
  
- **헤더 활용**: 데이터 형식(Content-Type), 인증 정보(Authorization) 등 부가 정보를 헤더에 담습니다.
  
- **무상태(Stateless)**: 서버는 클라이언트가 누군지 기억하지 않습니다. 매번 요청할 때마다 필요한 정보를 다 가져와야 합니다. (서버의 부담이 줄어들고 확장이 쉬워집니다.)

### 3. 리소스 조작을 위한 HTTP 메서드 (Action Verbs)

#### 메서드 | 역할 | 비유
- **GET**: 리소스 가져오기 (조회)  
  "그 책 좀 보여줘"
  
- **POST**: 리소스 새로 등록 (생성)  
  "새 책 한 권 사줘"
  
- **PUT**: 리소스 통째로 갱신 (수정)  
  "이 책을 새 버전으로 바꿔줘"
  
- **DELETE**: 리소스 삭제  
  "이 책 버려줘"

---

### 4. 상태 코드 (Status Codes): "서버의 대답"

#### 2xx (성공):  
- **200 OK**: "잘 됐어!"  
- **201 Created**: "잘 됐어! 새로 만들었어!"  

#### 4xx (클라이언트 에러):  
- **404 Not Found**: "너(손님)가 주문을 잘못했어!"  
- **400 Bad Request**: "너의 요청이 잘못됐어!"  

#### 5xx (서버 에러):  
- **500 Internal Server Error**: "미안, 내(가게)가 지금 문제가 생겼어!"

### 5. 스프링 MVC의 REST 지원 마법

스프링은 우리가 RESTful 서비스를 쉽게 만들도록 `HttpMessageConverter`를 제공합니다.

#### **HttpMessageConverter**  
자바 객체(Object)를 JSON 문자열로, 또는 JSON을 자바 객체로 자동으로 번역해 주는 **통번역기**입니다.

#### **@RestController**  
- **@Controller**와 **@ResponseBody**를 합친 것으로, 메서드가 리턴하는 값을 화면(HTML)이 아닌 **데이터(JSON)**로 바로 내보냅니다.

### 6. 핸들러 메서드 구현 예시

#### 💻 참조 계열(GET) API 만들기
```java
@RestController
@RequestMapping("/api/trainings")
public class TrainingRestController {
    private final TrainingService trainingService;

    public TrainingRestController(TrainingService trainingService) {
        this.trainingService = trainingService;
    }

    // 모든 교육 목록 조회 (JSON 리스트 반환)
    @GetMapping
    public List<Training> getTrainings() {
        return trainingService.findAll();
    }

    // 특정 ID로 교육 상세 조회
    @GetMapping("/{id}")
    public Training getTraining(@PathVariable String id) {
        return trainingService.findById(id);
    }

    // 헤더 참조 예시
    @GetMapping("/info")
    public String getInfo(@RequestHeader(HttpHeaders.USER_AGENT) String userAgent) {
        return "당신의 브라우저는: " + userAgent;
    }
}


@GetMapping("/custom/{id}")
public ResponseEntity<Training> getCustomTraining(@PathVariable String id) {
    Training training = trainingService.findById(id);
    // 상태 코드는 200 OK, 응답 헤더에는 커스텀 값 추가
    return ResponseEntity.ok()
            .header("X-Custom-Header", "Wook-Project")
            .body(training);
}
