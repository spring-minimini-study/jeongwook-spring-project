# 📚 Spring Framework 학습 기록 (1부: 7장)

> **핵심 주제**: JavaConfig 설정 방식과 `@Bean` 메서드의 활용

---

## 1. JavaConfig 클래스란?
- **개념**: XML 설정 파일 대신 자바 클래스를 사용하여 스프링의 설정 정보를 정의하는 방식입니다.
- **선언**: 클래스 상단에 `@Configuration` 애너테이션을 붙여 "이 클래스는 설정 파일이다"라고 선언합니다.

---

## 2. JavaConfig를 DI 컨테이너로 불러오는 3가지 방법

스프링이 어떤 설정 파일을 읽을지 결정하는 방법입니다. 실무에서는 보통 **컴포넌트 스캔** 방식을 가장 많이 사용합니다.

### ① AnnotationConfigApplicationContext 생성자 인수 방식
설정 클래스를 직접 나열하여 로드합니다.
```java
// FooConfig와 BarConfig를 각각 직접 지정하여 컨테이너 생성
ApplicationContext context = new AnnotationConfigApplicationContext(FooConfig.class, BarConfig.class);
```
### ② @Import 애너테이션 방식
여러 설정 파일을 하나의 메인 설정 파일로 계층화하여 묶을 때 사용합니다.
```java
@Configuration
@Import(BarConfig.class) // BarConfig의 설정 내용을 FooConfig에 합침
public class FooConfig {
// ...
}

// 실행 시에는 메인인 FooConfig만 불러오면 자동으로 BarConfig까지 로드됨
ApplicationContext context = new AnnotationConfigApplicationContext(FooConfig.class);
```
# ③ 컴포넌트 스캔 (@ComponentScan) 방식
특정 패키지 아래에 있는 모든 @Configuration 클래스를 자동으로 찾아 컨테이너를 구성합니다.
```java
@Configuration
@ComponentScan(basePackages = "com.jeongwook.config") // 해당 패키지 내 모든 설정 스캔
public class MainConfig { }
```
### 3. Bean 정의 방법: 스테레오타입 vs @Bean 메서드

빈을 만드는 두 가지 핵심 전략입니다. "누가 만든 클래스인가"에 따라 선택이 달라집니다.

* ✅ **스테레오타입 애너테이션 (`@Component`, `@Service` 등)**
 * **대상**: 내가 직접 코드를 작성한(Source 수정 가능한) 클래스.
 * **특징**: 클래스 위에 애너테이션만 붙이면 되어 매우 간결하며, 대량의 빈 등록에 유리합니다.
* ✅ **@Bean 메서드**
 * **대상**: 외부 라이브러리가 제공하는 클래스 (소스 수정이 불가능한 경우).
 * **특징**: `@Configuration` 클래스 내에서 메서드 형태로 직접 객체를 생성(`new`)하고 반환합니다.



```java
@Configuration
public class AppConfig {

    // 외부 라이브러리인 H2 데이터베이스의 DataSource를 빈으로 등록하는 예시
    @Bean
    public DataSource dataSource() {
        // 직접 객체를 조립(new)하여 스프링에게 전달
        return new EmbeddedDatabaseBuilder()
                .setType(EmbeddedDatabaseType.H2)
                .addScripts("schema.sql", "data.sql")
                .build();
    }
}
```

### 4. 빈 정의 방식 비교 요약

| 구분 | 스테레오타입 애너테이션 | @Bean 메서드 |
| :--- | :--- | :--- |
| **적용 위치** | 구상 클래스 상단 | JavaConfig 클래스 내 메서드 상단 |
| **주요 대상** | 개발자가 직접 만든 비즈니스 로직 클래스 | 라이브러리 제공 클래스 (DataSource 등) |
| **등록 방식** | 스캔을 통한 자동 등록 | 메서드 호출을 통한 수동 등록 |
| **권장 사항** | 일반적인 서비스, 리포지토리에 사용 | 복잡한 설정이 필요한 외부 객체에 사용 |

> XML 방식은 이제 레거시(구시대의 유물)이며, 현대적인 스프링 개발은 **애너테이션 기반 자동 등록**과 **JavaConfig를 통한 수동 등록**을 적절히 혼합하여 사용한다.

### 5. 의존성 주입(DI)의 정석: 생성자 인젝션

빈을 주입받을 때는 **생성자 인젝션**이 가장 안전하고 권장되는 방법입니다.

```java
@Service
public class TrainingServiceImpl implements TrainingService {
    
    // final 키워드로 객체의 불변성 확보 (런타임에 의존성이 바뀔 수 없도록 보장)
    private final TrainingRepository trainingRepository;

    // 생성자가 1개일 경우 @Autowired 생략 가능 (스프링 4.3 버전부터 지원)
    public TrainingServiceImpl(TrainingRepository trainingRepository) {
        this.trainingRepository = trainingRepository;
    }
}
```
