## 1. 트랜잭션(Transaction)이란?
- **정의**: 여러 개의 연관된 처리를 하나의 논리적인 작업 단위로 묶는 것을 말합니다.
- **원칙 (All or Nothing)**: 모든 작업이 성공하면 확정하고, 하나라도 실패하면 전체를 취소합니다.

### 💡 실생활 예시: 수강 신청
1. **SELECT**: 현재 수강 신청 인원 조회
2. **UPDATE**: 수강 신청 인원 수 +1 갱신
3. **INSERT**: 수강생 상세 정보 삽입

> **위험 상황**: 2번(갱신)은 성공했는데 3번(삽입)에서 에러가 난다면? 인원수만 늘어나고 명단은 없는 **데이터 불일치**가 발생합니다. 트랜잭션은 이를 방지합니다.

---

## 2. 트랜잭션의 핵심 제어: 커밋과 롤백

| 용어 | 설명 | 결과 |
| :--- | :--- | :--- |
| **커밋 (Commit)** | 모든 처리가 정상적으로 완료되었음을 확정하는 지시 | DB에 변경 사항이 영구적으로 저장됨 |
| **롤백 (Rollback)** | 처리 도중 오류가 발생하여 작업을 취소하는 지시 | 모든 데이터를 트랜잭션 시작 전 상태로 되돌림 |



---

## 3. 트랜잭션과 데이터베이스 커넥션(Connection)

데이터베이스 트랜잭션은 **'하나의 커넥션'** 안에서 이루어집니다.

- **주의점**: 여러 개의 Repository가 협력하더라도, 반드시 **동일한 커넥션**을 공유해야 하나의 트랜잭션으로 묶일 수 있습니다.
- **흐름**: 커넥션 획득 ➡️ 트랜잭션 시작 ➡️ 여러 SQL 실행 ➡️ 커밋/롤백 ➡️ 커넥션 반납

---

## 4. 트랜잭션 제어 방식 비교

### ❌ 수동 트랜잭션 제어 (레거시 방식)
코드가 복잡해지고 중복이 많아지며, 커넥션을 매번 파라미터로 전달해야 하는 번거로움이 있습니다.

```java
public void reserve(ReservationInput input) {
    Connection con = dataSource.getConnection();
    try {
        con.setAutoCommit(false); // 트랜잭션 시작 지시

        // 여러 리포지토리에 동일한 con을 전달해야 함
        trainingRepository.update(con, ...);
        reservationRepository.insert(con, ...);

        con.commit(); // 모든 성공 시 커밋
    } catch (Exception e) {
        con.rollback(); // 에러 발생 시 롤백
    } finally {
        con.close(); // 커넥션 닫기
    }
}
```
## 5. @Transactional 사용법 및 특징

스프링이 제공하는 **선언적 트랜잭션**은 복잡한 설정 없이 애너테이션 하나로 데이터의 안전성을 보장합니다.

### ✅ 주요 특징
* **메서드 단위 설정**: 특정 메서드에만 트랜잭션을 적용합니다. 해당 메서드가 호출될 때 트랜잭션이 열리고, 정상 종료 시 커밋됩니다.
* **클래스 단위 설정**: 클래스 레벨에 선언하면, 해당 클래스 내의 모든 `public` 메서드에 트랜잭션이 자동으로 적용됩니다.
* **롤백(Rollback) 대상**: 기본적으로 **비검사 예외(`RuntimeException` 및 그 하위 예외)**가 발생했을 때 자동으로 롤백을 수행합니다. (체크 예외는 별도 설정이 필요함)
* **간결성**: 개발자가 직접 `Connection`을 획득하거나 `setAutoCommit(false)`, `commit`, `rollback` 코드를 작성할 필요가 없어 비즈니스 로직에만 집중할 수 있습니다.



### 💻 활용 예시: 메서드 vs 클래스 설정

```java
// 1. 메서드 단위 설정: 필요한 곳에만 핀포인트로 적용
@Service
public class ReservationServiceImpl implements ReservationService {

    @Override
    @Transactional // 이 메서드 실행 시에만 트랜잭션 관리자가 개입함
    public void reserve(ReservationInput input) {
        // 비즈니스 로직 (조회, 수정, 삽입 등)
    }

    public void info() {
        // 트랜잭션이 필요 없는 단순 조회 메서드
    }
}

// 2. 클래스 단위 설정: 서비스 전체에 일관된 트랜잭션 정책 적용
@Service
@Transactional // 클래스 내의 모든 public 메서드에 트랜잭션이 적용됨
public class MemberServiceImpl implements MemberService {

    public void join(Member member) {
        // 트랜잭션 적용
    }

    public void leave(String memberId) {
        // 트랜잭션 적용
    }
}
```

##  스프링 트랜잭션 동작 원리 정리 (비유 중심)

---

### 1. Proxy(프록시) 객체의 정체: "연예인과 매니저"

`ReservationServiceImpl`이 실제 노래를 부르는 **연예인**이라면,  
스프링이 만든 **Proxy**는 그 연예인의 **매니저**

### 상황
- 팬(Controller)이 연예인에게 노래(메서드 실행)를 요청합니다.

### 스프링의 마법
- 스프링은 팬이 연예인을 직접 만나게 하지 않습니다.
- 대신 연예인과 똑같이 생긴 **매니저(Proxy)**를 앞에 세웁니다.

### Bean 등록
- `@Autowired`로 주입받는 객체는 실제 객체가 아니라  
  **Proxy(매니저) 객체**입니다.

---

### 🎤 호출 흐름 (예시 코드)

```java
// 1. 팬(Controller)이 요청을 보냄
public void reserveOrder() {
reservationService.reserve(...); // 사실은 Proxy(매니저)의 reserve 호출
}

// 2. Proxy(매니저)의 내부 동작
public class ReservationServiceProxy extends ReservationServiceImpl {
public void reserve(...) {
try {
// [사전 작업] 트랜잭션 시작
transactionManager.begin();

            // [실제 작업] 실제 서비스 호출
            super.reserve(...); 
            
            // [사후 작업] 커밋
            transactionManager.commit();
        } catch (Exception e) {
            // [예외 처리] 롤백
            transactionManager.rollback();
            throw e;
        }
    }
}
```

## 2. ThreadLocal: "개인용 마술 주머니"

"여러 Repository가 어떻게 같은 커넥션을 사용할까?"  
→ 그 답은 `ThreadLocal`

### 비유
- 헬스장의 **개인 사물함**

### 동작 방식
1. Proxy(매니저)가 DB에서 커넥션을 하나 가져옴
2. 해당 커넥션을 현재 Thread 전용 공간(ThreadLocal)에 저장
3. Repository는 파라미터 없이도 동일 커넥션 사용 가능

### 핵심 포인트
- "내 Thread의 저장 공간을 열어보면 항상 같은 Connection이 있다"

---

## 3. PlatformTransactionManager: "은행 지점장"

실제로 DB에 **커밋 / 롤백을 수행하는 핵심 객체**

### Bean 정의 예시
```java
@Bean
public PlatformTransactionManager transactionManager(DataSource dataSource) {
return new JdbcTransactionManager(dataSource);
}
```
### 역할
- 트랜잭션 시작 (`begin`)
- 커밋 (`commit`)
- 롤백 (`rollback`)

### 흐름
- Proxy → TransactionManager → DB

---

## 4. @EnableTransactionManagement: "매니저 고용 승인"
 
**"@Transactional이 붙은 객체에 Proxy를 자동으로 적용하겠다"**

### ComponentScan과의 차이
- `@ComponentScan`: Bean을 찾는 역할
- `@EnableTransactionManagement`: 트랜잭션 Proxy 적용 여부 결정
